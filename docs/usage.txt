aven usage
==========

Build:

  cargo build

Run checks:

  just check

Run the integration test suite:

  cargo test

Choose a local database path:

  aven --db /tmp/client-a.sqlite list
  AVEN_DB=/tmp/client-a.sqlite aven list

If neither is set, aven reads config from the platform config directory,
for example ~/.config/aven/config.yaml on Linux. If the
config has no local.db_path, aven stores data under XDG_STATE_HOME, or
~/.local/state when XDG_STATE_HOME is unset. The default database path is
~/.local/state/aven/db.sqlite.

Create a default config file:

  aven config init

Example config:

  local:
    db_path: "/tmp/client-a.sqlite"

  sync:
    enabled: true
    server_url: "http://127.0.0.1:3000"
    interval_seconds: 30

  daemon:
    wake_addr: "127.0.0.1:47631"

  project:
    overrides:
      - project: "Aven"
        paths: ["/path/to/agentic-task-manager"]

Project overrides map directories or repositories to project names for inferred
`aven add` tasks. They are useful when a directory basename differs from the
project name. `aven project create --path`, `aven project path add`, and
`aven project path remove` edit this config section. Git inference is used when
no override matches.

Common CLI examples:

  aven label create bug
  aven project create app --path /path/to/repo
  aven add "fix sync conflict display" --project app --label bug
  printf '## Context\nMarkdown works here\n' \
    | aven add "write docs" --project app --description-stdin
  aven list --project app
  aven list --deleted
  aven show APP-7KQ9 --full
  aven update 7KQ --status active --priority high
  printf 'handoff note\n' | aven note 7KQ --stdin
  aven delete 7KQ
  aven restore 7KQ
  aven doctor
  aven tui

TUI:

  aven tui

The TUI uses the same database selection rules as the CLI: --db, AVEN_DB,
then config. It opens a keyboard-first task browser with all tasks by
default, project-specific sidebar views, queue sorting, task detail, search,
manual refresh, and intent shortcuts for navigation, task actions, projects,
labels, views, filters, ordering, conflicts, and config.

Direct shortcuts cover navigation and app controls: j/k or arrows move,
Enter opens selection detail, / searches, : opens the command panel, ? opens
help, r refreshes, Tab switches focus, and q quits. Fast aliases include a for
add task, n for add note, s for status picker, d for done, and x for canceled.

Intent shortcuts use mnemonic prefix families. Press a prefix to see hints,
then press the next key:

  g  go to task list scope or switch workspace
  v  views such as queue, open, inbox, backlog, todo, active, done, conflicts
  f  filters such as label, priority, deleted visibility, or clear filters
  o  order by supported sort fields or reverse direction
  t  task fields and task lifecycle actions
  p  project administration
  L  label administration
  c  inspect and resolve conflicts
  C  show config, status, paths, or initialize config with confirmation

Esc cancels prefix mode and overlays. The command panel searches the same
catalog as the prefix hints and help overlay.

Sync server:

  aven server --bind 127.0.0.1:0 --data /tmp/aven-server.sqlite

The server prints:

  listening url=http://127.0.0.1:<port> scope=loopback

Bind scope is classified as loopback, private, or public. Loopback binds work
without authentication for local testing. Private binds (RFC1918, CGNAT
100.64.0.0/10, link-local, and unique-local IPv6) require sync.auth_token on
the server and matching client tokens. Public binds require
--unsafe-public-bind, sync.auth_token, and user-provided TLS or a reverse
proxy. The server prints a warning when a public bind is enabled.

Recommended VPN or private-network setup:

  # server config (~/.config/aven/config.yaml)
  sync:
    auth_token: "your-shared-secret"

  aven server --bind 100.64.0.10:3000 --data /tmp/aven-server.sqlite

  # client config
  sync:
    server_url: "http://100.64.0.10:3000"
    auth_token: "your-shared-secret"

Sync a client:

  aven --db /tmp/client-a.sqlite sync --server http://127.0.0.1:<port>
  aven sync

The second form uses sync.server_url from config, or AVEN_SYNC_SERVER if set.
The server stores an idempotent operation log. Clients push unsynced local
changes and pull changes after their local server cursor in bounded protocol
batches.

Run automatic sync for the configured local database:

  aven daemon

On macOS, install it as a user LaunchAgent:

  aven daemon install
  aven daemon uninstall

The daemon uses the same local SQLite database as the CLI. CLI commands still
work offline and write directly to SQLite. The daemon runs sync in the
background, wakes after successful local mutations when possible, and also
syncs periodically while it is running.

Known limitations:

  * Sync over plain HTTP is intended for loopback, a trusted VPN or private
    network, or TLS termination you provide externally. Direct public internet
    exposure requires --unsafe-public-bind and is otherwise unsupported.
  * Sync is an HTTP/JSON operation log, not a CRDT.
  * Sync supports one configured sync server per local database. Use a fresh
    database for a different sync server.
  * Path mappings are local to one device.
  * There is no web UI, Mac app, GitHub polling, due date, relation, custom
    status, recurring task, or team feature.
  * Conflict resolution is explicit and field based.
