atm usage
=========

Build:

  cargo build

Run checks:

  just check

Run the integration test suite:

  cargo test

Choose a local database path:

  atm --db /tmp/client-a.sqlite list
  ATM_DB=/tmp/client-a.sqlite atm list

If neither is set, atm reads config from the platform config directory,
for example ~/.config/agentic-task-manager/config.toml on Linux. If the
config has no local.db_path, atm stores data under the operating system app
data directory, for example ~/.local/share/atm/db.sqlite on Linux.

Create a default config file:

  atm config init

Example config:

  [local]
  db_path = "/tmp/client-a.sqlite"

  [sync]
  enabled = true
  server_url = "http://127.0.0.1:3000"
  interval_seconds = 30

  [daemon]
  wake_addr = "127.0.0.1:47631"

Common CLI examples:

  atm label create bug
  atm project create app --path /path/to/repo
  atm add "fix sync conflict display" --project app --label bug
  printf '## Context\nMarkdown works here\n' \
    | atm add "write docs" --project app --description-stdin
  atm list --project app
  atm show APP-7KQ9 --full
  atm update 7KQ --status active --priority high
  printf 'handoff note\n' | atm note 7KQ --stdin
  atm delete 7KQ
  atm restore 7KQ
  atm tui

TUI:

  atm tui

The TUI uses the same database selection rules as the CLI: --db, ATM_DB,
then config. It opens a keyboard-first task browser with all tasks by
default, project-specific sidebar views, queue sorting, task detail, search,
manual refresh, and triage shortcuts for status, priority, delete, and
restore.

Sync server:

  atm server --bind 127.0.0.1:0 --data /tmp/atm-server.sqlite

The server prints:

  listening url=http://127.0.0.1:<port>

Sync a client:

  atm --db /tmp/client-a.sqlite sync --server http://127.0.0.1:<port>
  atm sync

The second form uses sync.server_url from config, or ATM_SYNC_SERVER if set.
The server stores an idempotent operation log. Clients push unsynced local
changes and pull changes after their local server cursor.

Run automatic sync for the configured local database:

  atm daemon run

The daemon uses the same local SQLite database as the CLI. CLI commands still
work offline and write directly to SQLite. The daemon runs sync in the
background, wakes after successful local mutations when possible, and also
syncs periodically while it is running.

Known limitations:

  * v1 has no authentication.
  * v1 sync is an HTTP/JSON operation log, not a CRDT.
  * v1 supports one configured sync server per local database. Use a fresh
    database for a different sync server.
  * Path mappings are local to one device.
  * There is no web UI, Mac app, GitHub polling, due date, relation, custom
    status, recurring task, or team feature.
  * Conflict resolution is explicit and field based.
